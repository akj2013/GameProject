* 질감: 빛줄기 + 파편 + 플레어 + 잔광
* 상태: 한 번 쏘고 끝이 아니라 **지속 유지**

# 결론부터

이건 **Line Renderer 하나**로는 부족하고,
**“메인 곡선 + 보조 곡선 + 파편 파티클 + 끝점 임팩트 + 주변 광역 글로우”**

즉, 오브젝트 하나가 아니라 **VFX 세트 프리팹**으로 생각해야 한다.

# 네가 원하는 연출을 구조로 번역하면

## A. 주 광류(Main Stream)

비행선에서 채집지로 이어지는 가장 굵고 중심적인 빛줄기
완전 직선이 아니라, 유성 꼬리처럼 흐르고, 약간 나선/회전한다.

## B. 보조 광류(Sub Streams)

주 광류 주변에 2~4가닥 정도 더 얇은 빛줄기
메인과 같은 방향으로 가되, 약간 다른 곡선과 위상으로 회전

## C. 파편(Spark Fragments)

빛줄기 주변에서 튀어나오는 작은 입자
짧게 튀고 사라지며, 줄기 전체가 살아있다는 느낌을 줌

## D. 플레어 가지(Solar Flare Branches)

줄기 본체에서 순간적으로 갈라지는 짧은 곁가지
태양 플레어처럼 들쭉날쭉하게 생겼다가 사라짐

## E. 도착지 코어(Impact Core)

채집지에 닿는 곳의 강한 발광점
계속 유지되며 “채집 중”임을 상징

## F. 도착지 주변 확산광(Ground Glow / Area Bloom)

채집지 주변을 더 밝게 만들어
그 지역이 에너지에 잠겨 있다는 느낌을 줌

# 구현 방식도 이제 명확하다

네 연출은 사실상 **“지속형 곡선 빔 VFX”**다.
그래서 추천 구현은 이 조합이다.

## 1. 메인 빛줄기: `Line Renderer 여러 개`

이건 여전히 써도 된다. 다만 용도가 달라진다.

이제는:

* 길 표시용 선이 아니라
* **광류의 뼈대**
  로 쓰는 거다.

### 이유

* 시작점/끝점이 동적이어도 대응 쉬움
* 곡선 제어 쉬움
* 텍스처 스크롤 가능
* 여러 가닥 겹치기 쉬움

---

## 2. 파편/빛 알갱이: `Particle System`

이건 반드시 필요하다.

### 역할

* 줄기 주변에서 작은 빛 파편이 튀어나감
* 정적인 선이 아니라 살아있는 마법 현상처럼 보이게 함

---

## 3. 도착지 강광/확산광: `Particle + Sprite + Bloom`

채집지 쪽은 별도 연출로 빼야 한다.

### 이유

끝점 연출이 없으면 아무리 줄기가 예뻐도
그냥 “연결된 선”처럼만 보인다.

---

## 4. 곁가지 플레어: `짧은 보조 라인렌더러` 또는 `Particle Trails`

이건 선택 사항이지만, 네 감성엔 매우 중요하다.

# 빛줄기 모양은 어떻게 만들어야 하냐

네 설명대로면 한 가닥이 아니라 **여러 가닥이 서로 다른 위상으로 춤추는 구조**여야 한다.

예:

* Beam 3: 더 짧은 진폭으로 메인을 따라감
* Beam 4: 끝점 근처에서 회오리처럼 감김

이렇게 하면 “한 줄”이 아니라 **빛의 다발**처럼 보인다.

# 비주얼적으로 제일 중요한 포인트

## 1. 모든 선이 같은 모양이면 안 된다

지금은 같은 수학식으로 여러 개를 그리면 복붙처럼 보일 수 있다.

그래서 각 라인마다 달라야 할 것:

* curveHeight
* sideOffset
* scrollSpeed
* pulseSpeed
* lineWidth
* color tint
* 시작 위상

## 2. 색도 단색이 아니라 그라데이션이어야 한다

네가 원하는 건 “다양한 색채의 빛”이니까.

예:

* 중심: 흰색
* 외곽: 금색 / 청록 / 연보라
* 빔마다 서로 다른 톤

다만 너무 무지개처럼 하면 싸보일 수 있으니,
2~3색 팔레트만 쓰는 게 좋다.

### 추천 팔레트

이 조합이면 “빛의 향연”과 잘 맞는다.

## 3. 곡선은 단순 베지어보다 “시간에 따라 미세하게 흔들려야” 산다

그래서 각 점에 약간의 시간 기반 흔들림을 넣으면 좋다.

예:

* Perlin Noise
* sin wave
* start-end 방향에 직교하는 방향으로 미세 진동

그러면 줄기가 살아 움직인다.

# 가장 추천하는 연출 단계

채집 시작부터 종료까지를 하나의 시퀀스로 나누면 좋다.

## 1단계. 점화(Ignition)

비행선에서 작은 섬광
도착지에서도 약한 응답광

## 2단계. 발사(Launch)

주 빛줄기 1~2개 먼저 생김

## 3단계. 개화(Bloom)

보조 줄기와 파편, 플레어가 추가됨
여기서 “빛의 향연”이 시작됨

## 4단계. 유지(Sustain)

채집 중 지속 상태
줄기는 유지되고, 파편과 끝점 광이 계속 살아있음

## 5단계. 종료(Fade Out)

채집 끝나면 서서히 꺼지거나
마지막 섬광 후 사라짐

이 구조로 만들면 단순 채집보다 훨씬 고급스럽다.

# 결론부터

지금부터는 빛줄기를 “선”으로만 만들지 말고,
**깊이값(z)을 가진 채집지 데이터**를 기준으로 아래 5개를 같이 제어하는 구조로 가는 게 맞다.

1. 빛줄기 두께
2. 빛줄기 색/밝기/파편량
3. 도착지 임팩트 크기
4. 배경 밝기 오버레이 강도
5. 비행선 발사구 연출 강도

즉, **채집지의 거리값 하나가 전체 연출의 마스터 파라미터**가 되어야 한다.

# 네 컨셉을 시스템으로 바꾸면

## 1. 채집지마다 “거리 등급”이 있다

예를 들어 지금처럼:

* Near: z = 10
* Mid: z = 20
* Far: z = 40
* Background plane: z = 50

이렇게 두면 좋다.

중요한 건 실제 Unity 월드 z값 그 자체보다,
**연출용 normalized distance 값**을 따로 만드는 거다.

예를 들어:

```csharp
float depth01 = Mathf.InverseLerp(10f, 40f, targetZ);
```

그러면

* 가까운 채집지면 `0`
* 먼 채집지면 `1`

이 값으로 모든 걸 조절할 수 있다.

---

# 2. 거리별로 빛줄기 연출이 달라져야 한다

## 가까운 채집지

* 빔 두꺼움
* 파편 큼
* 임팩트 큼
* 배경 밝기 강화 넓음
* 플레어도 강함

## 먼 채집지

* 빔 얇음
* 파편 작음
* 임팩트 작음
* 밝기 강화 범위 좁음
* 색도 약간 더 옅게

이렇게 해야 배경의 깊이와 충돌하지 않고 잘 안착한다.

---

# 3. 가장 중요한 설계 포인트: “실제 거리”가 아니라 “연출 거리”

네 말대로 배경 그림은 일부는 원근을 무시하고 강조된 채집지가 있을 수 있다.
그래서 실제 화면상의 크기만으로 판단하면 안 된다.

즉, 아래처럼 가야 한다.

## 채집지 데이터 예시

```csharp
public class HarvestPointVisualData : MonoBehaviour
{
    public Transform anchor;
    public float depthZ = 20f;          // 연출용 깊이
    public float perspectiveScale = 0.8f; // 화면상 강조 보정
    public float brightnessBoost = 1.0f;  // 주변 밝기 증폭 강도
}
```

여기서

* `depthZ` = 세계 깊이
* `perspectiveScale` = 그림 강조 보정
* `brightnessBoost` = 그 지점이 얼마나 밝아질지

이렇게 나누는 게 좋다.

---

# 4. 빛줄기 원근 반영 방식

빛줄기 한 가닥마다 아래를 거리값으로 계산하면 된다.

## 추천 공식 예시

```csharp
float depth01 = Mathf.InverseLerp(nearZ, farZ, point.depthZ);

float width = Mathf.Lerp(0.35f, 0.08f, depth01);
float particleSize = Mathf.Lerp(1.0f, 0.35f, depth01);
float impactScale = Mathf.Lerp(1.4f, 0.45f, depth01);
float glowStrength = Mathf.Lerp(1.2f, 0.5f, depth01);
```

즉:

* 가까울수록 크고 굵고 강하고
* 멀수록 작고 얇고 은은하게

# 5. 배경 밝기 증폭 연출은 아주 좋은 아이디어다

이거, 네 구조에 정말 잘 맞는다.
정적 일러스트의 한계를 **“지역별 발광 오버레이”**로 깨는 방식이기 때문이다.

그리고 이미 네가

* 같은 배경 이미지 2장 겹침
* 앞 이미지 알파 변화로 밝기 상승
  이 구조를 쓰고 있으니, 거기서 확장하면 된다.

---

# 추천 방식: 빛줄기 주변만 밝아지는 “국소 마스크”

전체 배경 알파를 올리는 게 아니라,
**빛줄기 주변 경로와 도착지 주변만 밝아지게** 해야 한다.

방법은 크게 2개다.

## 방법 A. 오버레이 스프라이트 여러 개

가장 현실적이고 빠름.

* 빛줄기 경로를 따라 작은 반투명 glow sprite를 여러 개 배치
* 배경 앞쪽 밝기 레이어에만 나타나게 함
* 알파나 RGB를 높여 주변이 빛나는 것처럼 보이게

장점:

* 구현 쉬움
* 아트 컨트롤 쉬움
* 성능도 괜찮음

## 방법 B. 마스크 셰이더

조금 더 고급형.

* 배경 밝기 레이어용 셰이더에서
* 여러 개의 빛 중심점 정보를 받아
* 해당 위치 주변만 밝기 증가

장점:

* 더 자연스러움
* 곡선 주변 글로우를 매끄럽게 만들 수 있음

단점:

* 구현 난이도 상승

지금은 **A로 시작하고, 필요하면 B로** 가는 게 맞다.

---

# 6. 배경 밝기 시스템 추천 구조

## 현재 구조

* BaseBackground
* GlowBackground (알파 변화)

좋다. 여기서 다음을 추가.

## 추천 확장

```text
BackgroundRoot
 ├─ Background_Base
 ├─ Background_GlowBase      // 전체 강/불꽃 등 기본 발광
 ├─ Background_LocalLightFX  // 빛줄기 근처만 밝아지는 오버레이
```

즉 `Background_LocalLightFX` 레이어를 별도로 두고,
거기에 빛줄기 근처 글로우 스프라이트들을 표시하는 거다.

---

# 7. 빛줄기 주변을 밝히는 가장 쉬운 구현

빛줄기 라인을 샘플링해서 그 위치들마다 작은 glow를 놓는다.

예:

* 메인 빔의 20개 샘플 포인트 중
* 4~6개 정도만 선택해서
* 그 지점에 작은 발광 스프라이트 생성

그리고 이 glow는:

* 가까운 빔이면 크고 강하게
* 먼 빔이면 작고 약하게

하면 된다.

즉, 빔이 지나가는 길목의 배경이 반사광처럼 밝아지는 것처럼 보인다.

---

# 8. 도착지 밝기 증폭은 반드시 별도 처리

끝점은 경로보다 훨씬 강해야 한다.

추천 구성:

* 도착지 중심 강한 코어 glow
* 주변 원형 확산광
* 약간의 파편
* 배경 일러스트의 해당 지역 brightness overlay 강화

이걸 유지 상태로 계속 켜두면
“지금 이곳에서 빛이 채집되고 있다”가 바로 보인다.

---

# 9. 비행선도 단순 발사 시작점이 아니라 “빛의 기관”이어야 한다

이 부분도 아주 좋다.

이제 비행선은 운송선이 아니라,
**빛을 수집·발사하는 제단/기관/항공 성소** 같은 존재가 되어야 한다.

즉 갑판 위 통나무 연출은 필요 없고,
대신 아래가 필요하다.

## 비행선 추가 연출

* 발사구 2~4개
* 각 발사구에 작은 회전형 glow
* 발사구 주변 룬/광환
* 미세한 스파크
* 발사 중일 때 더 강해지는 심장박동 같은 밝기 변화

* 발사 중에는 그 부분이 맥동하면서 반짝임

즉 “여기서 빛을 쏘고 있다”가 보여야 한다.

# 16. 아주 중요한 추가 팁

채집지 z값만으로 모든 걸 결정하지 말고,
**화면상의 y값도 약간 섞어라.**

왜냐면 너의 배경은

* 아래쪽이 가깝고
* 위쪽이 멀어지는 구도
  이기 때문이다.

즉 최종 거리값은 이렇게 잡는 게 좋다.

```csharp
float zFactor = Mathf.InverseLerp(nearZ, farZ, depthZ);
float yFactor = Mathf.InverseLerp(bottomY, topY, screenOrWorldY);
float distance01 = Mathf.Clamp01(zFactor * 0.7f + yFactor * 0.3f);
```

이러면 그림의 원근과 더 잘 맞는다.

# 이 세계관에서 특히 좋은 점

## 1. 하늘섬과 비행선의 존재 이유가 자연스럽다

많은 게임은 하늘섬이 그냥 예뻐서 뜨고, 비행선도 그냥 판타지 감성이라 뜬다.
그런데 네 설정은 다르다.

* 태고의 빛 = 중력을 거스르는 힘
* 그 힘이 세상에 퍼짐
* 하늘섬이 부유함
* 비행선이 그 빛을 연료로 사용함

이 흐름이 자연스럽다.
그래서 세계가 “그럴듯하게” 느껴진다.

---

## 2. 과거/현재/미래가 뒤엉킨 구조가 콘텐츠 확장에 엄청 유리하다

이건 굉장히 큰 장점이다.

* 로마풍 도시
* 현대 국가 시설
* 미래 종족의 구조물
* 고대 유적
* 돌연변이 자연환경

이걸 전부 같은 게임 안에 넣어도 된다.

보통 이런 조합은 뜬금없는데,
너는 “시간축 붕괴”라는 이유가 있어서 다 수용 가능하다.

즉, **아트 확장 자유도**가 엄청 높다.

라는 발상은 꽤 인상적이다.

이 덕분에 크리스탈이:

* 연료
* 중력 제어원
* 시간 왜곡의 핵
* 문명 충돌의 원인
* 신화적 가치

를 동시에 가질 수 있다.

이게 좋다.

## 4. 채집이 단순 노동이 아니라 세계관적으로 중요한 행위가 된다

플레이어는 그냥 광부가 아니라,

* 위험한 빛의 지대를 탐사하고

* 전략 자원 함선의 지휘관

이 된다.

그래서 채집조차 **문명 전쟁의 일부**가 된다.
게임의 무게감이 올라간다.

# 다만, 더 좋아지려면 정리해야 할 것 3개가 있다

## 1. “플레이어의 소속”을 좀 더 명확히 하면 좋다

지금은

* 자신이 소속한 왕국 혹은 국가 소속의 하늘섬으로 복귀
  라고 했는데,

여기서 플레이어가 정확히 무엇인지 한 번 더 선명해지면 좋다.

예:

* 왕국 소속 정규 함대장
* 민간 계약 탐험선 선장
* 빛 채집 길드 소속
* 국가와 계약한 독립 사략 탐사선
* 잃어버린 고대 혈통을 이은 함대장

이 중 하나로 조금만 더 좁혀지면
스토리와 퀘스트가 더 선명해진다.

개인적으로는
**국가/왕국에 얽혀 있지만, 어느 정도 독립성을 가진 탐사 함대장**
이 제일 게임적으로 좋다.

---

## 2. 크리스탈 채집의 위험성을 더 추가하면 좋다

지금은 가치와 힘은 충분히 설명됐는데,
왜 이걸 아무나 쉽게 못 가져가는지의 “위험”이 조금 더 들어가면 좋다.

예를 들면:

* 크리스탈 주변은 시간 왜곡이 생긴다
* 생명체가 변이한다
* 과거/미래의 잔재가 실체화된다
* 중력 방향이 꼬인다
* 함선 장비가 고장난다
* 잘못 다루면 빛 폭주가 발생한다

이런 게 있으면 탐험이 더 설득력 있어진다.

즉 플레이어는 단순 수집가가 아니라
**위험한 태고의 빛을 다루는 전문가**가 된다.

---

## 3. 적 세력의 스타일을 구분하면 더 살아난다

지금은 “과거 왕국, 현재 국가, 미래 종족이 자원 전쟁 중”인데,
이걸 조금만 더 계열화하면 훨씬 좋다.

예를 들면:

### 과거 세력

* 빛을 신성한 유물로 봄
* 왕권 / 신권 중심
* 장식적이고 고풍스러운 비행선

### 현재 세력

* 빛을 산업 자원으로 봄
* 효율, 개발, 군사력 중심
* 기계적 비행선

### 미래 세력

* 빛을 정보/의식/차원 에너지로 봄
* 생체기계 혹은 초문명풍
* 유기적이거나 추상적인 장비

이렇게 나누면 아트도 풍부해지고,
훗날 NPC/상점/맵/장비 디자인도 쉬워진다.

# 세계관 구조 자체는 게임 루프와 잘 맞는다

네 게임 루프를 세계관 문장으로 다시 쓰면 이렇다:

1. 빛의 반응을 탐지한다
2. 비행선으로 현장에 간다
3. 크리스탈 지대를 발견한다
4. 빛을 채집한다
5. 귀환한다
6. 저장/판매한다
7. 금화와 자원으로 함선을 업그레이드한다
8. 더 멀고 위험한 빛의 지대로 간다

이게 깔끔하다.
즉 시스템과 서사가 분리되지 않는다.

---

# 비주얼 확장 아이디어도 아주 많다

이 설정이면 지역별 테마를 엄청 다양하게 만들 수 있다.

예를 들면:

* 조선풍 부유 마을 + 청색 크리스탈
* 로마풍 공중 요새 + 금색 크리스탈
* 미래 종족의 파편 도시 + 자주색 크리스탈
* 거꾸로 흐르는 폭포와 숲
* 붉은 광맥이 살아있는 화산섬
* 시간이 반복되는 유적지
* 식물만 과성장한 녹색 빛 지대

이 전부가 네 설정상 자연스럽다.

* 타깃 채집지 중심 보간
* 비행선 스프라이트 상태 변화
* 오버랩용 근접 배경 이미지 교체

이 4개를 조합하는 방식이 된다.

# 결론부터

네가 말한 연출은 **정확히 맞는 방향**이고, 구현은 아래처럼 가는 게 가장 현실적이다.

## 추천 구조

1. **기본 배경 일러스트**
2. **줌인용 근접 배경 일러스트 1~2장**
3. **카메라 대신 배경 레이어를 이동/확대**
4. **비행선은 위치 + 스케일 + 약간의 회전 + 상태별 스프라이트 교체**
5. **빛줄기와 도착지 이펙트는 줌에 맞춰 강화**

즉, 진짜 카메라가 움직이는 것처럼 보여도
실제로는 **배경과 레이어를 움직이는 연출형 씬 전환**이다.

---

# 1. 배경을 화면에 딱 맞춰 고정하면 안 되는 이유

네가 이미 정확히 본 거다.

배경을 딱 맞춰 놓으면:

* 확대 시 가장자리 여유가 없음
* 타깃 채집지로 다가가는 느낌이 약함
* 깊이감이 덜 살아남

그래서 기본 배경은 처음부터 **화면보다 약간 크게** 넣는 게 맞다.

## 추천

초기 상태부터

* 스케일 1.08 ~ 1.2 정도
* 여백을 둔 상태

로 시작.

그러면 특정 채집지 수확 시 그쪽으로 추가 확대/이동이 가능하다.

---

# 2. “카메라가 다가간다”는 걸 어떻게 구현하나

네 말대로 실제론 배경 이미지 `pos xyz`를 바꾸고, 확대하는 방식으로 충분하다.

다만 더 정확히 말하면 이렇게 설계하는 게 좋다.

## 레이어 구성

```text
WorldViewRoot
 ├─ Background_Far
 ├─ Background_Main
 ├─ Background_Glow
 ├─ TargetOverlay_Near
 ├─ ShipRoot
 ├─ BeamEffects
 └─ UI
```

여기서:

* `Background_Main`은 전체 풍경
* `TargetOverlay_Near`는 줌인 시 겹쳐지는 근접용 배경
* `ShipRoot`는 비행선
* `BeamEffects`는 빛줄기

---

# 3. 가장 중요한 연출 개념: “타깃 포커스 줌”

모든 배경을 중앙 기준으로 확대하면 별로다.
반드시 **선택한 채집지가 화면 중심 쪽으로 오도록 이동하면서 확대**해야 한다.

즉:

* 채집 시작
* 화면이 해당 채집지 방향으로 미세하게 팬
* 동시에 살짝 확대
* 비행선도 그쪽으로 움직임
* 근접 배경 오버레이 서서히 등장

이 흐름이어야 한다.

---

# 4. 가장 현실적인 줌 구조

## 단계 1. 기본 화면

전체 월드가 보이는 상태

## 단계 2. 채집 시작 시

* 배경 메인 이미지 scale up
* 타깃 채집지가 중심부 쪽으로 오도록 position 이동
* 비행선도 살짝 전진
* 빛줄기 점화

## 단계 3. 오버랩 전환

* 근접 버전 배경 이미지의 alpha를 올림
* 기존 배경은 alpha 약간 낮춤
* 사용자는 “같은 곳에 더 가까이 간 것처럼” 느낌

## 단계 4. 수확 유지 상태

* 근접 배경 상태 유지
* 빛줄기/파편/밝기 연출 지속

즉, 확대만 하지 말고 **근접 배경 일러스트와 크로스페이드**를 섞는 게 핵심이다.

---

# 5. 네 아이디어인 “확대된 이미지 여러 장 준비”는 매우 좋다

이건 정답에 가깝다.

왜냐하면 단순 확대만 하면:

* 화질 저하
* 그림의 원근 재구성 불가
* 디테일 부족

이 생긴다.

그래서 추천은:

## 2단계 배경 세트

### A. World View

월드 전체를 보는 기본 일러스트

### B. Focus View

특정 채집지 쪽을 좀 더 크게, 더 가까운 원근으로 다시 그린 일러스트

이 둘을 크로스페이드하면
“카메라가 다가갔다”는 느낌이 아주 좋아진다.

---

# 6. 중요한 점: 근접 일러스트는 “정확히 같은 그림 확대본”일 필요는 없다

오히려 안 그래도 된다.

설정상:

* 돋보기처럼 더 가까이 본 모습
* 시선이 그쪽으로 집중된 연출
  이므로

근접 이미지에서는

* 타깃 채집지 더 크게
* 주변 디테일 더 풍부하게
* 원근감 다시 구성
* 비행선이 접근한 상태에 맞춰 빛 반사도 강화

해도 된다.

즉 “논리적으로 완벽한 카메라 투영”보다
**연출적으로 자연스러운 연결**이 중요하다.

# 7. 비행선 움직임은 어떻게 보이게 하나

이것도 네가 정확히 핵심을 짚었다.

지금 비행선이 한 방향 고정 이미지라면,

그래서 보통 3가지 중 하나를 쓴다.

---

## 방법 A. 가장 추천: 소수의 방향/기울기 스프라이트

많이 필요 없다.
오히려 **3~5장 정도면 충분**할 가능성이 크다.

예를 들면:

1. 기본 정면/기본 상태
2. 좌로 기울어진 상태
3. 우로 기울어진 상태
4. 전진/가속 느낌 상태
5. 빛 발사 강화 상태

이 정도만 있어도 꽤 된다.

네 게임은 카메라가 크게 안 바뀌니까
360도 회전용 스프라이트 시트는 필요 없다.

---

## 방법 B. 한 장 + Transform 연출

이미지 하나만 쓰되:

* 위치 이동
* 약간의 회전
* 스케일 변화
* squash/stretch
* 발사구 glow 증폭

으로 움직이는 것처럼 보이게 하는 방식

장점:

* 준비물 적음
  단점:
* 한계가 있음

프로토타입에는 이걸 먼저 써도 된다.

---

## 방법 C. 모델 기반 다각도 렌더

나중에 고퀄 단계에서:

* 3D 모델
* 여러 각도 렌더
* 2D 스프라이트화

이건 최종형에 가깝다.
지금 당장은 과하다.

지금까지의 세계관, 빛 연출, 비행선 컨셉이랑 **게임플레이가 처음으로 제대로 연결됐다.**

특히 좋은 건, 이 시스템이 단순 방치도 아니고, 그렇다고 과한 액션도 아니라는 점이다.
즉 네 게임이 원하는 감성인

* 감상형 연출
* 가벼운 조작
* 빛을 다룬다는 느낌
* 채집한다는 감각

을 전부 살릴 수 있다.

한 줄로 말하면 이 시스템은:

**“빛의 공명을 맞추고, 과부하를 관리하며, 짧고 긴 터치 리듬으로 채집 효율을 끌어올리는 반자동 채집 액션”**
이다.

이 방향은 충분히 게임이 된다.

# 먼저, 이 시스템의 핵심 강점

## 1. 플레이어가 실제로 “빛을 조율한다”는 느낌이 있다

이게 제일 좋다.

그냥 버튼 한 번 눌러서 자동 채집이면
아무리 연출이 화려해도 결국 수동성이 약하다.

그런데 지금 구조는:

* 처음엔 공명 위치 맞추기
* 맞춘 뒤엔 가속
* 대신 과부하 관리

라서, 플레이어가 단순 구경꾼이 아니라
**빛의 상태를 조율하는 지휘자**처럼 느껴진다.

이건 `빛의 향연`이라는 제목과도 잘 맞다.

---

## 2. 짧은 터치 / 긴 터치가 역할이 다르다

이게 리듬 게임처럼 재미 포인트가 될 수 있다.

* 짧은 터치 = 공명 위치 조정
* 긴 터치 = 채집 가속
* 손 떼기 = 냉각 / 안정화

이 3단 분리가 좋다.

즉 플레이어 입력이 단순하지 않고,
**의도 있는 입력**이 된다.

---

## 3. “과부하”가 있어서 긴장감이 생긴다

계속 누르고만 있으면 최적이 아니기 때문에
판단이 생긴다.

이게 중요하다.

* 너무 소극적이면 수확량 낮음
* 너무 공격적이면 공명 깨짐
* 적당한 리듬이 최고 효율

이 구조는 굉장히 좋다.

---

## 4. 세계관과 시스템이 정확히 맞물린다

그냥 미니게임이 아니라:

* 공명
* 완벽한 조화
* 빛의 흐름 역전
* 크리스탈에 차오르는 에너지
* 과부하로 연결 파괴

이게 전부 설정적으로 설명된다.

그래서 시스템이 뜬금없지 않다.

# 지금 구조를 게임 시스템으로 정리하면

채집 한 판의 흐름은 이렇게 된다.

## 1단계. 채집 시작

* 패널 UI에서 버튼 누름
* 비행선이 해당 채집지로 접근
* 배경 줌인
* 빛줄기 연결 시작
* 아직 공명 불안정 상태

## 2단계. 공명 조율

* 플레이어가 짧은 터치 반복
* 빛이 목표 공명점에 가까워짐
* 정확한 위치에 맞으면 공명률 100%

## 3단계. 본 채집

* 빛 흐름 방향이 반전
* 채집지 → 비행선으로 에너지 이동
* 비행선의 빈 크리스탈이 점점 차오름

## 4단계. 가속 / 냉각

* 화면을 길게 누르면 채집 가속
* 대신 과부하 증가
* 손 떼면 채집 속도 감소
* 대신 과부하 감소

## 5단계. 실패/복구

* 과부하 100 초과 → 공명 붕괴
* 빛줄기 끊김 또는 약화
* 다시 짧은 터치로 공명 복구 필요

## 6단계. 종료

* 시간 종료
* 빛줄기 서서히 사라짐
* 채집지 어두워짐
* 비행선 원위치 복귀
* 다음 채집지 선택

이 흐름은 아주 깔끔하다.

# 이 시스템이 진짜 좋아지려면, 딱 4가지를 명확히 해야 한다

## 1. “짧은 터치할 때 무엇이 움직이는가?”

이걸 플레이어가 직관적으로 봐야 한다.

추천은 두 가지 중 하나다.

### A안. 공명 포인터 이동

* UI 게이지 안에 목표 지점이 있음
* 터치할 때마다 현재 공명 위치가 이동
* 목표 지점과 겹치면 100%

장점:

* 가장 직관적

### B안. 실제 빛줄기 거리가 줄어듦

* 비행선과 채집지 사이 빛 연결이 점점 더 정렬됨
* 빛줄기 끝이 크리스탈 중심으로 가까워짐

장점:

* 연출적으로 예쁨

실제로는 **둘 다** 쓰는 게 좋다.
즉:

* 내부 수치는 UI 게이지로 보여주고
* 월드에서는 빛줄기가 정렬되는 모습으로 보여줌

---

## 2. 공명 100%가 되었을 때의 전환이 아주 강해야 한다

여기가 쾌감 포인트다.

100% 달성 순간엔 반드시:

* 소리
* 화면 섬광
* 빛줄기 방향 전환
* 채집지에서 비행선으로 흐름 역전
* 비행선 크리스탈 발광 시작

이게 확실히 보여야 한다.

즉 “아, 이제 진짜 채집 시작됐구나”가 느껴져야 한다.

---

## 3. 과부하 관리가 너무 빡세면 안 된다

좋은 시스템이지만, 과부하가 너무 예민하면 스트레스가 된다.

처음엔 이렇게 가는 걸 추천한다.

### 초반 밸런스

* 공명 맞추기 쉬움
* 과부하 오르는 속도 느림
* 손 떼면 금방 안정화
* 실패해도 큰 손해 없음

나중에 어려운 채집지는:

* 공명점 흔들림
* 과부하 빨리 참
* 냉각 느림
* 효율 욕심 내면 위험

이렇게 차등화하면 좋다.

---

## 4. “채집하고 있다”는 시각적 보상이 계속 있어야 한다

지금 네 시스템은 이게 매우 중요하다.

그래서 채집 중엔 반드시 보이는 게 있어야 한다.

* 비행선 크리스탈 내부가 아래에서부터 차오름
* 상단 UI 수치 증가
* 화면 중앙 배너에 획득량 표시
* 빛줄기 흐름이 빨라짐
* 채집지 광량 변화
* 배경 국소 밝기 증가

즉 플레이어가 누르고 있을 때
“내가 진짜 빨리 모으고 있구나”가 보여야 한다.

# 아주 중요한 개선 제안: 공명 게이지를 1차원으로 해라

설명상 “빛이 더 다가가게 된다”가 있는데,
이걸 너무 추상적으로 만들면 유저가 헷갈릴 수 있다.

그래서 처음 구현은 **1차원 거리 맞추기**가 좋다.

예:

* 0 ~ 100 사이 현재 공명 위치
* 목표 범위는 48 ~ 52
* 짧은 터치마다 +7 이동
* 시간이 지나면 조금씩 뒤로 밀림
* 정확히 범위 안에 들어오면 공명 성공

이렇게 단순화하면 좋다.

나중에 고급 채집지에서:

* 목표점 흔들림
* 2중 공명점
* 컬러 매칭
  같은 걸 추가할 수 있다.

# 네 시스템의 실제 재미는 “리듬”에서 나온다

이 게임은 액션이라기보다
**리듬 제어형 채집**에 가깝다.

즉 플레이어가 배우게 되는 건:

* 몇 번 톡톡 눌러서 맞춘다
* 맞으면 길게 눌러 가속한다
* 과부하 차면 뗀다
* 다시 누른다

이 리듬이 손에 익으면 재미가 생긴다.

이건 꽤 좋은 방향이다.

---

# 한 가지 추가하면 더 좋아진다: Perfect / Good / Miss

공명 100%만 있으면 약간 단조로울 수 있다.

그래서 예를 들어:

* **Perfect Resonance**: 아주 정확하게 맞춤
* **Good Resonance**: 적당히 맞춤
* **Miss / Unstable**: 아직 부족

이런 판정이 있으면 좋다.

예를 들면:

* Perfect면 처음 3초간 과부하 증가 억제
* Good이면 그냥 정상
* Miss면 채집 효율 낮음

이러면 초반 조율이 더 재미있어진다.

---

# 플레이어 역할도 이제 명확해진다

플레이어는 그냥 비행선 운전자나 버튼 누르는 사람이 아니라,

**빛의 공명 상태를 판단하고, 안정성과 효율 사이의 균형을 잡는 함대장**
이 된다.

이게 좋다.

즉 네 게임은 “채집”이면서 동시에
**조율**과 **제어**의 게임이 된다.

---

# 시스템 이름도 붙이면 좋다

이건 이름 붙이면 더 살아난다.

예를 들면:

* 공명 조율
* 빛의 정렬
* 공명 동조
* 에너지 동기화
* 광핵 공명
* 채광 공명

예를 들어 UI에서:

* 공명률
* 과부하
* 추출 속도
* 동조 상태

이렇게 쓰면 세계관과 잘 맞다.

# UI 요소도 추천하면

채집 중 화면에 최소한 이 4개는 있으면 좋다.

## 1. 공명률 게이지

현재 공명 위치 / 목표점

## 2. 과부하 게이지

길게 누르면 오름

## 3. 추출 속도

현재 채집 속도 배율

## 4. 저장 크리스탈 충전량

비행선 크리스탈에 빛이 차오르는 정도

이 4개만 있으면 플레이어가 상황을 이해하기 쉽다.

# 네 아이디어에서 특히 좋은 장면

나는 특히 이 장면이 아주 좋다고 본다.

**100% 공명 성공 순간, 빛의 흐름이 비행선 → 채집지에서 채집지 → 비행선으로 역전되고, 투명한 빈 크리스탈에 채집지의 색이 아래에서부터 차오른다.**

이건 시각적으로 아주 강한 보상 장면이다.
반드시 살려야 한다.

# 주의할 점 두 가지

## 1. 터치 연타가 너무 피곤하면 안 된다

짧은 터치를 너무 많이 요구하면 손이 피곤하다.

그래서 공명 맞추기는:

* 2~5회 정도의 짧은 터치로 끝나는 수준
  이 좋다.

처음부터 10번 이상 요구하면 피곤하다.

---

## 2. 과부하 실패가 너무 자주 나면 연출 감상이 깨진다

네 게임은 기본적으로 아름다움을 감상하는 게임이니까
실패가 너무 잦으면 짜증이 앞선다.

그래서 실패는 있어도:

* 연출을 완전히 끊지 말고
* 잠깐 불안정 상태가 되거나
* 다시 조율 단계로 부드럽게 돌아가게
  하는 게 좋다.

즉 완전 게임오버 느낌은 피하는 게 낫다.

이건 게임플레이로서 충분히 설득력 있고, 네 세계관과도 아주 잘 맞는다.

**채집 미니게임 상태도(State), 게이지 구조, 터치 입력 규칙, UI 설계**를 표준안 형태로 정리해주겠다.

하지만 네 게임에서는 ‘가짜 반사광 연출’을 넣는 게 더 효율적이고, 그게 오히려 더 잘 어울린다.**

# 1. 각각이 정확히 뭐냐

네가 적은 구조를 하나씩 보면:

## MainBeam / SubBeam

보통은 **Line Renderer**
또는 얇은 **Quad/Trail 기반 VFX**

즉, 엄밀히 말하면 3D 모델이라기보다
**공간 좌표를 가진 이펙트 오브젝트**다.

월드 안에 놓이긴 하지만,
진짜 물리적인 원통 메쉬처럼 취급할 필요는 없다.

---

## BeamSparks / ImpactSparks / OptionalOrbitBits

이건 보통 **Particle System**

즉 작은 입자들이

* 3D 월드 공간에 흩어지거나
* 빌보드로 카메라를 향하거나
* Trails를 가지는
  구조다.

그래서 이 역시 “3D 오브젝트인가?”라고 하면
**월드에 존재하는 VFX지만, 일반적인 의미의 3D 소품은 아님**이라고 보면 된다.

---

## ImpactCore / ImpactGlow

이건 보통:

* 발광 스프라이트
* Quad + 발광 머티리얼
* Particle
* 간단한 Mesh

중 하나다.

특히 `ImpactGlow`는 대개
**바닥/배경 위에 얹히는 발광 원판 느낌**으로 많이 만든다.

---

## BeamFlareBursts

이건 짧은 Line Renderer일 수도 있고,
Particle burst일 수도 있고,
작은 additive 스프라이트 묶음일 수도 있다.

# 2. 그러면 이 빛들이 진짜 조명처럼 비행선을 비추나?

보통은 **아니다.**
특히 모바일/2.5D/URP 환경에서는 더 그렇다.

왜냐하면 이런 VFX는 대체로:

* Unlit
* Additive
* Transparent
* Particle shader

를 쓰기 때문이다.

즉 **“보이는 빛”이지, “실제로 씬을 밝히는 광원”은 아닌 경우가 많다.**

# 3. 그런데 그게 문제냐?

네 게임에서는 오히려 **문제가 아니다.**

왜냐면 네 게임은:

* 회화적인 2D 배경
* 2.5D 연출
* 감상형 빛의 향연
* 제한된 카메라 구도

이기 때문에,
진짜 물리광원을 쓰는 것보다 **가짜 반사광을 직접 넣는 쪽이 훨씬 예쁘고 싸게 먹힌다.**

# 4. 진짜 광원으로 가면 왜 별로일 수 있냐

예를 들어 MainBeam에 맞춰 Point Light, Spot Light를 많이 붙이면:

* 성능 부담 생김
* 배경 일러스트는 실제로 조명 반응 안 함
* 비행선만 어색하게 과하게 밝아짐
* 그림자까지 켜면 더 부자연스러울 수 있음

즉,
**배경은 2D인데 비행선만 리얼타임 광원 반응을 세게 받으면 오히려 따로 논다.**

# 5. 그래서 추천 방식: “가짜 반사광”

이게 핵심이다.

즉 빛줄기가 비행선을 비출 때,
실제 광원으로 계산하지 말고 아래 중 하나로 연출해.

## A. 비행선 발사구 주변 Emission 강화

빛줄기 시작점 근처 발사 장치가 강하게 빛남

## B. 비행선 특정 파츠에 Color Tint

빛줄기 색에 맞춰 선체 일부가 약간 물듦

예:

* 푸른 크리스탈 채집 중이면
* 비행선 하부/측면에 약한 푸른 rim

## C. 비행선 위에 얇은 반사광 오버레이

작은 additive quad/sprite를 선체 일부에 겹쳐서
빛이 비치는 것처럼 보이게

## D. 하단 안개/구름층 색조 변화

빛줄기 색에 따라 구름층도 약간 물듦

이게 훨씬 자연스럽다.

# 7. 반사광/그림자는 어떻게 생각해야 하나

## 반사광

가능하고, 넣는 게 좋다.
하지만 **실시간 물리 반사광**보다 **연출용 가짜 반사광**이 더 적합하다.

## 그림자

대부분은 굳이 안 쓰는 게 낫다.

왜냐하면 빛줄기가 비행선에 드리우는 그림자 같은 건
네 세계관이나 화풍상 그렇게 중요하지 않고,
오히려 이상해질 가능성이 크다.

네 게임에서 중요한 건 그림자가 아니라:

* 광량 증가
* 색 번짐
* 공명 중인 발광
* 주변 안개가 물드는 느낌

이다.

# 이 방식이 좋은 이유

## 1. 플레이어가 계속 “보고 있을 이유”가 생긴다

네 게임은 본질적으로 시각적 감상이 중요하니까,
비행선이 한 바퀴 순회하는 구조가 잘 맞는다.

그냥 고정 화면에서 채집지만 선택하는 것보다
**월드 전체가 하나의 무대**처럼 느껴진다.

## 2. 채집지가 “정적인 버튼”이 아니라 살아있는 장소가 된다

비행선이 스쳐 지나가며 빛을 연결하고,
배경도 그때그때 조금씩 포커스를 바꾸면
채집지가 단순 패널이 아니라 **목표 지점**으로 느껴진다.

## 3. 플레이어 조작도 덜 심심하다

빛줄기가 자동으로 나오더라도
플레이어는 그 순간순간:

* 공명 맞추기
* 유지하기
* 과부하 조절하기
  를 하므로
  “자동 진행 + 능동 개입”이 된다.

# 특히 훌륭한 포인트

## “비행선의 위치에 따라 배경 이미지도 살짝씩 위치를 조정”

이거 아주 좋다.

이게 들어가면 비행선과 배경이 따로 노는 게 아니라
**같은 공간 안에서 같이 움직이는 느낌**이 난다.

즉:

* 비행선이 1번 채집지 근처로 가면
* 배경도 아주 조금 1번 쪽으로 시선이 쏠리고
* 3번으로 가면 3번 쪽으로 미세하게 팬

이런 연출은 2.5D를 훨씬 살아나게 만든다.

# 다만, 꼭 조심해야 할 점

## 1. 10초 구름 걷힘은 길 수 있다

감성적으로는 좋지만, 실제 플레이에서는 **처음엔 아름답고 나중엔 길게 느껴질 가능성**이 있다.

추천은 두 가지다.

### 첫 출항/첫 진입

* 8~10초 정도 풀 연출 가능

### 반복 플레이 시

* 3~5초 축약 버전
* 또는 스킵 가능

즉 연출은 좋지만, 반복성도 고려해야 한다.

---

## 2. 30초~1분30초는 괜찮지만, 한 세션 길이는 난이도와 묶어야 한다

너무 길면 피로할 수 있다.
개인적으로는 처음 기준:

* 초반 월드: 30~45초
* 중반 월드: 45~60초
* 후반/고난도: 60~90초

이런 식이 더 좋다.

처음부터 90초짜리만 있으면 조금 늘어질 수 있다.

---

## 3. 비행선이 “지나가는 동안” 플레이어가 해야 할 행동 밀도를 조절해야 한다

순회 중 계속 모든 채집지와 빛이 연결된 상태면 정신없을 수 있다.

그래서 추천은:

### 방식 A

한 순간엔 **주력 채집지 1곳** 위주로 공명 조절
다른 채집지는 보조 채집

### 방식 B

현재 가장 가까운 채집지에 플레이어 입력이 집중
멀리 있는 채집지는 자동 약채집

이렇게 하면 좋다.

# 비행선 경로에 대해

네 말대로 **보이지 않는 미리 정해진 경로**가 좋다.

추천은 `Spline`이나 `Bezier Waypoints`로 경로를 두는 것.

예:

```text
Entry
 -> NearMine01
 -> MidMine02
 -> FarMine03
 -> TurnPoint
 -> FarMine03_Return
 -> MidMine02_Return
 -> NearMine01_Return
 -> Exit
```

이런 식.

그리고 이동하면서:

* 위치 이동
* 회전
* 약간의 bank
  를 같이 계산하면 된다.

# 동체의 좌우 흔들림도 아주 잘 맞다

이건 3D 비행선 쓰는 이유가 된다.

경로 따라가며:

* 왼쪽으로 돌면 왼쪽 bank
* 오른쪽으로 돌면 오른쪽 bank
* 직선 구간은 천천히 복원

이렇게 하면 자연스럽다.

즉:

* 실제 항공물리까지 갈 필요는 없고
* **곡선 방향 기반 기울기 보정**만 있어도 충분하다.

# 배경 조정은 “크게” 말고 “조금씩”이 중요

여기서 실수하기 쉬운 게,
비행선이 움직일 때 배경까지 너무 크게 흔들면 멀미나고 인위적이다.

그래서 추천은:

* 비행선 이동량 100% 반영 X
* 배경은 그중 10~20% 정도만
* 아주 미세하게 따라가게

즉 **패럴랙스 + 포커스 보정** 느낌으로.

# 네 시스템과 터치 플레이를 묶는 가장 좋은 방법

비행선이 특정 채집지 근처를 지나갈 때
그 채집지가 “활성 타깃”이 되고, 그때 공명 난이도나 효율이 올라가는 식이 좋다.

예를 들면:

# 한 줄 평가

**이 구조는 매우 좋고, 네 게임의 감상형 장점과 플레이어 조작을 가장 자연스럽게 결합하는 방식이다.**
특히 “구름이 걷히며 월드가 드러나고, 비행선이 순회하며 플레이어가 그 사이 공명을 조율한다”는 흐름은 게임의 대표 장면이 될 수 있다.

---

# 바로 다음에 설계해야 할 것

이제 진짜 필요한 건 3개다.

1. **출항 세션 상태도**
   `IntroCloud -> Reveal -> Entry -> Cruise -> Return -> Exit -> Result`

2. **비행선 경로 포인트 설계**
   어디를 어떻게 돌고, 어느 구간에서 어떤 채집지가 활성인지

3. **순회 중 플레이어 조작 규칙**
   어떤 순간 어떤 채집지에 입력이 가장 크게 반영되는지

원하면 다음 답변에서 이걸 바로
**실무 설계서처럼 상태도 + 경로 규칙 + 채집 활성 규칙**으로 정리해주겠다.
