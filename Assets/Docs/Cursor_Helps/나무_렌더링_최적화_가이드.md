# Tree_PineGreen / PineWinter / PineYellow 렌더링 최적화 가이드

## 현재 프리팹 구조 요약

| 프리팹 | 메시 | 머티리얼(서브메시당 1개) | Static |
|--------|------|--------------------------|--------|
| Tree_PineGreen | mesh guid `28ec0bb...` | 2개 (고유) | 0 (비활성) |
| Tree_PineYellow | **동일 메시** `28ec0bb...` | 2개 (Green과 **다름**) | 0 |
| Tree_PineWinter | **다른 메시** (nested prefab) | 2개 (Winter 전용) | 0 |

- 나무 하나당 **MeshRenderer 1개, 서브메시 2개** → **드로우콜 2개/나무**.
- Green과 Yellow는 **메시는 같고 머티리얼만 다름** (색만 다름).
- Winter는 **메시·머티리얼 모두 다름**.
- 세 프리팹 모두 **Static 체크 없음** (쓰러지므로 비정적이 맞음).

---

## 1. Static Batching이란?

- **의미**: 움직이지 않는 오브젝트들을 Unity가 **한 번만** “합쳐서” 그리도록 하는 기능.
- **조건**:  
  - Hierarchy에서 **Static** 체크.  
  - **같은 머티리얼**을 쓰는 오브젝트끼리만 한 배치로 묶임.  
  - 빌드/플레이 시 메시가 합쳐져서 **드로우콜 수가 줄어듦**.
- **당신 프로젝트에선**:  
  - 나무는 **쓰러지고 비활성화**되므로 **움직이는(변하는) 오브젝트**에 가깝습니다.  
  - Static을 켜면 움직임/SetActive 등과 충돌할 수 있어 **나무에는 비추천**.  
  - **도로/타일/건물** 같은 진짜 안 움직이는 것에만 Static + Static Batching 사용하는 게 좋습니다.

**정리**: 나무에는 Static Batching **적용하지 않는 것**이 좋음.

---

## 2. GPU Instancing이란?

- **의미**: **같은 메시 + 같은 머티리얼**인 오브젝트를 GPU가 **한 번의 드로우콜**로 여러 개 그리게 하는 기능.
- **동작**:  
  - CPU는 “이 메시 + 이 머티리얼로, 위치/스케일/회전만 다른 N개 그려줘” 한 번만 보냄.  
  - GPU가 내부에서 인스턴스별 행렬로 여러 번 그림 → **드로우콜 수 감소**, CPU 부담 감소.
- **조건**:  
  - **메시 1종류 + 머티리얼 1종류** 조합당 한 배치.  
  - 셰이더가 **GPU Instancing 지원**이어야 함 (Unity URP Lit 등 대부분 지원).  
  - 머티리얼에서 **Enable Instancing** 체크.
- **당신 나무에 적용하려면**:  
  - Green / Yellow는 **메시가 같고** 색만 다름 → **색을 머티리얼 프로퍼티(예: _BaseColor)로 넣고**, 런타임에 **MaterialPropertyBlock**으로 색만 바꾸면 **같은 머티리얼 1개**로 인스턴싱 가능.  
  - Winter는 **메시가 다르므로** Green/Yellow와는 별도 배치. Winter끼리만 인스턴싱 가능.

**정리**: Green/Yellow는 “메시 통일 + 머티리얼 1개 + 색만 MaterialPropertyBlock”으로 **GPU Instancing 적용하는 것이 가장 효과적**. Winter는 같은 Winter끼리만 인스턴싱.

## 3. Frame Debugger로 Draw Call 수 확인하는 방법

- **의미**: **매 프레임** 카메라가 “무엇을, 몇 번 그렸는지” 단계별로 보여주는 Unity 에디터 도구.
- **용도**:  
  - “나무가 많을 때 드로우콜이 몇 개로 늘어나는지” 직접 확인.  
  - 배칭/인스턴싱 적용 전·후 비교.
- **사용 절차**:  
  1. **Window → Analysis → Frame Debugger** 로 창 연다.  
  2. **Play** 누른 뒤, 나무가 많이 보이는 상황으로 카메라 이동.  
  3. Frame Debugger에서 **Enable** 눌러서 캡처.  
  4. 왼쪽 트리에서 **Camera.Render** 펼치면 **Draw Xxx**, **Draw Mesh**, **Draw Mesh Instanced** 같은 항목이 나열됨.  
  5. **같은 이름/머티리얼이 반복되는 수** = 대략적인 드로우콜 수.  
  6. 나무 머티리얼 이름으로 검색해서 “나무 때문에 몇 번 그려지는지” 센다.
- **해석**:  
  - 예: 나무 100그루 × 서브메시 2 = 200개 드로우가 나오면 부담이 큼.  
  - Instancing 적용 후에는 “Draw Mesh Instanced” 한두 개로 합쳐지는지 확인하면 됨.

**정리**: “나무 많을 때” 씬에서 Frame Debugger 켜고, Draw/Draw Mesh 개수 보고 → 최적화 후 다시 켜서 비교하는 흐름이 좋음.

## 4. 제안했던 5가지 — 이 3종 프리팹 기준으로 정리

### ① 드로우콜 줄이기 (머티리얼 통일 + 배칭/인스턴싱)

- **현재**: Green/Yellow/Winter가 **서로 다른 머티리얼** → 종류별로 따로 그려짐. 인스턴싱/배칭 거의 안 됨.
- **추천**:  
  - **Green / Yellow**:  
    - 메시는 이미 같음.  
    - “나무용 머티리얼 1개”만 두고, 색은 **셰이더의 _BaseColor(또는 _Color)** 같은 걸로 받도록 한 뒤,  
    - 스폰/초기화 시 **MaterialPropertyBlock**으로 색만 넣어 주기.  
    - 머티리얼에서 **Enable GPU Instancing** 체크.  
  - **Winter**:  
    - Winter 전용 머티리얼 1개로 통일하고, Instancing 켜면 Winter끼리만 인스턴싱.
- **효과**: 나무 수가 많을 때 **드로우 수가 크게 줄어** FinishFrameRendering 비용 감소에 가장 도움이 됨. **우선 적용 권장.**

---

### ② 화면에 나오는 나무 수 줄이기 (Cull / LOD)

- **현재**: CameraDistanceCuller로 cull 21, enable 20 사용 중.
- **추천**:  
  - **Cull Distance를 더 줄이기** (예: 18~15): 화면에 동시에 보이는 나무 수를 줄이면 드로우 수가 그만큼 줄어듦.  
  - **LOD Group**: 멀리 있는 나무는 저폴리/빌보드로 바꿔서 폴리곤·드로우 비용 감소 (구현 난이도는 있음).
- **효과**: “나무가 많이 보일 때”의 **최대 드로우 수**를 줄이는 데 좋음. ①과 함께 쓰면 좋음.

---

### ③ 그림자·라이트 비용 줄이기

- **현재**: 나무가 그림자 캐스터로 쓰이고 있을 가능성 높음.
- **추천**:  
  - URP **Shadow Distance** 줄이기.  
  - 나무 머티리얼/렌더러에서 **Cast Shadows = Off** 시험 (나무 그림자가 꼭 필요 없다면).  
  - 필요하면 **Shadow Resolution** 낮추기.
- **효과**: FinishFrameRendering 안에서 **그림자 관련 CPU/GPU 비용**을 줄일 수 있음. ①, ② 다음으로 적용해 보면 좋음.

---

### ④ 오버드로우 줄이기

- **의미**: 같은 픽셀을 여러 번 그리는 것. 나무 잎이 겹치면 픽셀 셰이더가 여러 번 실행됨.
- **추천**:  
  - 나무용 셰이더가 **불필요하게 무겁지 않은지** 확인.  
  - 가능하면 **나무 전용 라이트웨이트 셰이더** 하나로 통일 (Lit, Single Pass 등).
- **효과**: GPU 픽셀 연산이 줄어들 수 있음. ①~③ 적용 후에도 부하가 크면 검토.

---

### ⑤ 스크립트/업데이트 부담

- **현재**: TreeManager, CameraDistanceCuller 등이 나무마다 붙어 있음. Culler는 이미 0.2초 간격.
- **추천**:  
  - 나무 수만큼 **매 프레임** 도는 Update가 있는지 확인하고, 필요하면 **몇 프레임에 한 번**만 돌리거나 범위 제한.  
  - 이미 Culler는 가벼우므로, **우선순위는 ①~④**.
- **효과**: Script 쪽 비용이 클 때 보조적으로 의미 있음.

---

## 5. 이 3종 프리팹에 “무엇이 좋은지” 요약

| 순서 | 항목 | 이 3종 나무에 적합 여부 | 비고 |
|------|------|-------------------------|------|
| 1 | **GPU Instancing** (Green/Yellow 머티리얼 1개 + MaterialPropertyBlock 색) | ✅ 매우 적합 | 메시 이미 같음, 색만 다름. 효과 가장 큼. |
| 2 | **Cull Distance 줄이기** | ✅ 적합 | 설정만으로 화면 나무 수 감소. |
| 3 | **그림자 거리/해상도 조정, Cast Shadows Off 시험** | ✅ 적합 | URP/라이트 설정 + 렌더러 옵션. |
| 4 | **LOD Group** | △ 선택 | 멀리 나무 단순화 시 유리. 작업량 있음. |
| 5 | **Static Batching** | ❌ 비추천 | 나무는 비정적(쓰러짐)이라 맞지 않음. |

**실전 추천 순서**:  
1) **Frame Debugger**로 나무 드로우 수 확인 →  
2) **Green/Yellow 머티리얼 1개 + MaterialPropertyBlock + GPU Instancing** 적용 →  
3) **Cull Distance** 소폭 감소 →  
4) **그림자** 설정 조정.  

이후 다시 Frame Debugger로 드로우 수가 줄었는지 확인하면, “나무가 많이 들어올 때 렉”이 어디서 줄었는지 바로 보입니다.
