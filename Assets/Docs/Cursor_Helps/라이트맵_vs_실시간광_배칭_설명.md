# 라이트맵 vs 실시간 방향광 — 배칭에 미치는 영향

## 1. 실시간 방향광만 쓸 때 → 배칭 조건이 단순한 이유

### 어떻게 그리는가
- **방향광 1개**: 씬 전체에 같은 방향/색/강도의 빛이 적용됨.
- 셰이더는 **전역 데이터**만 사용:
  - `_MainLightPosition`, `_MainLightColor` (또는 URP의 Light 버퍼)
  - 그림자는 Shadow Map **1장** (또는 캐스케이드 몇 장) — 모든 오브젝트가 **같은 텍스처**를 참조.

### 배칭이 잘 되는 이유
- **머티리얼이 같으면** (같은 셰이더, 같은 텍스처, 같은 설정):
  - 오브젝트 A, B, C… 모두 **동일한 셰이더 변수 + 동일한 텍스처 세트**를 사용.
- GPU 입장에서는:
  - “이 패스는 이 머티리얼로, 이 텍스처들로 그리면 된다” → 한 번 설정하고 여러 메쉬를 연속으로 그림.
- 그래서 **SRP Batcher / GPU Instancing**이 같은 머티리얼끼리 잘 묶어 줌.

**요약:** 실시간 방향광은 **오브젝트마다 다른 데이터를 바인딩하지 않음** → 머티리얼만 같으면 배칭 조건이 단순함.

---

## 2. 라이트맵을 쓰면 → 배칭이 잘리는 이유

### 라이트맵이란
- **베이크된 조명**을 **텍스처**로 저장해 두고, 렌더링할 때 그 텍스처를 샘플링해서 씀.
- 씬에 있는 오브젝트들을 **라이트맵 아틀라스** 한 장(또는 여러 장)에 “붙여 넣기”해서:
  - 오브젝트 A → 아틀라스의 (0.1, 0.2)~(0.3, 0.4) 영역  
  - 오브젝트 B → (0.5, 0.1)~(0.7, 0.3)  
  - … 이런 식으로 **오브젝트마다 다른 UV 영역**을 가짐.

### 셰이더 입장에서
- 라이트맵을 쓰는 셰이더는 보통:
  - **공통**: `unity_Lightmaps` (라이트맵 아틀라스 텍스처)
  - **오브젝트마다 다름**: `unity_LightmapST` — “이 오브젝트는 아틀라스의 **어느 구역(UV 스케일/오프셋)**을 쓸지”
- 즉, **같은 머티리얼**이어도:
  - Draw Call 1: 머티리얼 + LightmapST = (타일1 영역)
  - Draw Call 2: 머티리얼 + LightmapST = (타일2 영역)
  - Draw Call 3: …

### 왜 배칭이 깨지는가
- **SRP Batcher**는 “같은 셰이더 + **같은 per-renderer 데이터**”일 때만 한 배치로 묶음.
- 라이트맵을 쓰면 **Renderer마다** `unity_LightmapST`(라이트맵 UV)가 다름 → per-renderer 데이터가 다름.
- 따라서 “머티리얼은 1개”여도 **오브젝트마다 다른 라이트맵 영역**을 쓰기 때문에 배치가 나뉘어 버림.
- 추가로, 라이트맵용 셰이더 변형(키워드)이 있으면 머티리얼/패스가 더 쪼개질 수 있음.

**요약:** 라이트맵 = **오브젝트마다 “다른 텍스처 영역(UV)”을 쓰는 것과 같은 효과** → per-object 데이터가 달라서 배칭이 잘림.

## 3. 현재 씬 설정 (SampleSceneTest 1 등)

프로젝트 씬 파일을 기준으로 한 **현재 상태**입니다.

| 항목 | 값 | 의미 |
|------|-----|------|
| **m_EnableBakedLightmaps** | 1 | **베이크 라이트맵 사용**으로 설정됨. |
| **m_EnableRealtimeLightmaps** | 0 | 실시간 라이트맵(Realtime GI)은 꺼져 있음. |
| **m_Lightmapping** (렌더러) | 4 | 오브젝트들이 **라이트맵/GI에 기여**하도록 설정됨 (Contribute GI). |
| **m_ScaleInLightmap**, **m_StitchLightmapSeams** | 있음 | 라이트맵 베이크 시 **UV/차트** 사용 준비된 상태. |
| **m_LightingDataAsset** | 기본 참조 | 씬에 연결된 라이트맵 데이터는 기본값일 수 있음 (실제 베이크 여부는 Unity에서 확인 필요). |

### 정리
- **설정상**: “베이크 라이트맵을 쓰는” 방향으로 되어 있고, 오브젝트들도 라이트맵용으로 설정돼 있음.
- **실제로 Window → Rendering → Lighting에서 Bake를 한 적이 있다면**  
  → 셰이더가 라이트맵을 읽도록 되어 있을 때 **오브젝트마다 다른 라이트맵 UV**가 들어가므로, 위에서 말한 대로 **배칭이 잘리는 구조**가 됨.
- **한 번도 Bake를 안 했다면**  
  → 라이트맵 텍스처가 없을 수 있어, 현재는 실시간광만 쓰는 것에 가깝을 수 있음. (에디터에서 Lighting 창으로 확인하는 것이 정확함.)

## 4. 배칭을 최대한 살리고 싶을 때

- **드로우콜/배칭 우선**이면:
  - **실시간 방향광만** 쓰고, 라이트맵 베이크를 쓰지 않거나 끄면 배칭이 가장 단순해짐.
  - 같은 머티리얼(PandaMat 등)만 쓰고, 실시간 그림자만 사용하는 구성이 유리함.
- **비주얼(간접광, 풍부한 그림자) 우선**이면:
  - 라이트맵을 쓰되, “배칭이 잘린다”는 비용을 감수하는 식으로 가져가면 됨.
  - 모바일에서는 라이트맵 해상도를 낮추거나, 라이트맵을 쓰는 오브젝트 수를 줄이는 식으로 타협할 수 있음.

## 5. Unity에서 현재 씬이 어떻게 되어 있는지 확인하는 방법

1. **Window → Rendering → Lighting** 열기.
2. **Scene** 탭에서:
   - **Baked Global Illumination** / **Realtime Global Illumination** 체크 여부.
   - **Generate Lighting** 버튼으로 베이크를 한 적이 있는지.
3. **Environment** 탭:
   - **Source**에 **Lighting Data Asset**이 할당돼 있는지 (할당돼 있으면 해당 씬은 베이크된 라이트맵 사용 중).
4. 씬에서 아무 **MeshRenderer** 하나 선택 → Inspector:
   - **Lighting** 섹션에 **Lightmap Index** / **Lightmap Scale Offset** 등이 채워져 있으면, 그 오브젝트는 **실제로 라이트맵을 쓰는 중**.

이렇게 보면 “실시간 방향광 위주인지, 라이트맵이 실제로 쓰이고 있는지”를 정확히 알 수 있고, 그에 따라 “배칭이 단순한지 / 라이트맵 때문에 잘리는지”를 판단할 수 있습니다.
